<objective>
Scaffold the vendor app's 6-step onboarding wizard and 4-tab shell. After login a chef is routed to the wizard (if onboarding not complete), the wizard saves each step to the API, and submitting the final step lands the chef on a "pending review" holding screen. Once admin approves (status: verified) the root layout routes to tabs.

Purpose: Unblocks all subsequent vendor feature screens which live inside (tabs). The onboarding gate is the entry control that prevents unverified chefs from reaching dashboard/orders/menu.
Output: Working 6-step onboarding, pending/rejected holding screen, 4-tab shell stub screens.
</objective>

Onboarding gate logic (add AFTER existing auth check):
```typescript
const { data: onboardingStatus, isLoading: onboardingLoading } = useQuery({
  queryKey: ['chef', 'onboarding', 'status'],
  queryFn: () => api.get<OnboardingStatusResponse>('/chef/onboarding/status'),
  enabled: isAuthenticated && !isLoading,
  staleTime: 60_000,
});

useEffect(() => {
  if (isLoading || onboardingLoading) return;
  if (!isAuthenticated) {
    router.replace('/(auth)/login');
  } else if (!onboardingStatus) {
    return; // still fetching
  } else if (onboardingStatus.data.status === 'verified') {
    router.replace('/(tabs)');
  } else if (onboardingStatus.data.status === 'not_started') {
    router.replace('/(onboarding)/personal-info');
  } else {
    // pending_review, submitted, rejected, in_progress
    router.replace('/(onboarding)/pending');
  }
}, [isAuthenticated, isLoading, onboardingStatus, onboardingLoading]);
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| onboarding form → API | User-controlled fields (business name, hours, phone) cross into API |
| device camera/gallery → API | Binary file uploads (images, PDFs) cross into multipart endpoint |
| JWT → onboarding status endpoint | Token from SecureStore must be valid to fetch status |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01-01 | Tampering | POST /chef/onboarding body | mitigate | All fields validated with Zod on client before submit; server enforces its own validation — client validation provides user feedback, not security |
| T-03-01-02 | Spoofing | Root layout onboarding gate | mitigate | Gate reads from API (not local storage) — status 'verified' only set by admin server-side; cannot be bypassed by modifying local state |
| T-03-01-03 | Information Disclosure | Document upload (ID proof, FSSAI) | mitigate | Images uploaded via authenticated multipart POST with Bearer token; URIs stored only in Zustand memory (not persisted to disk in plain text) |
| T-03-01-04 | Elevation of Privilege | Reaching (tabs) before verified | mitigate | Onboarding gate checks API-side status on every app launch; even if router.replace is called manually, tab screens call RequireChef endpoints which return 403 |
| T-03-01-05 | Denial of Service | 30s polling on pending screen | accept | Low frequency (30s); no PII in polling; minor battery cost accepted for v1 |
</threat_model>
