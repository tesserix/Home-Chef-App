<!-- Go API shapes for vendor menu and secondary screens -->

GET /api/v1/chef/menu response:
```typescript
interface MenuResponse {
  items: MenuItem[];
  categories: Category[];
}
interface MenuItem {
  id: string;
  name: string;
  description: string;
  price: number;
  category: string;
  isAvailable: boolean;
  isVeg: boolean;
  images: { id: string; url: string }[];
  preparationTime: number;  // minutes
}
```

POST /api/v1/chef/menu/items body: `{ name, description, price, category, isVeg, preparationTime }`
Returns: `{ item: MenuItem }` — use returned `item.id` for subsequent image upload.

PUT /api/v1/chef/menu/items/:itemId — same fields; price change creates ApprovalRequest on server.

POST /api/v1/chef/menu/items/:itemId/images — multipart/form-data field `file` (image/jpeg or image/png, max 5MB per image).

GET /api/v1/chef/analytics?period=week|month|year response:
```typescript
interface AnalyticsResponse {
  period: string;
  totalOrders: number;
  totalRevenue: number;
  popularItems: { name: string; orders: number; revenue: number }[];
  dailyRevenue: { date: string; revenue: number }[];
}
```

GET /api/v1/chef/reviews response:
```typescript
interface ReviewsResponse {
  reviews: { id: string; customerName: string; rating: number; comment: string; createdAt: string; reply?: string }[];
  averageRating: number;
  totalReviews: number;
}
```

POST /api/v1/chef/reviews/:reviewId/reply body: `{ reply: string }`

GET /api/v1/chef/payout response:
```typescript
interface PayoutResponse {
  bankAccount: { bankName: string; accountNumber: string; ifsc: string } | null;
  upiId: string | null;
  totalEarnings: number;
  pendingPayout: number;
  lastPayout: { amount: number; date: string } | null;
  earningsHistory: { week: string; amount: number }[];
}
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| menu item form → API | Price, name, description fields cross from user input to API |
| review reply → API | Free-text reply field crosses from chef input to API (stored and shown to customers) |
| profile photo → GCS | Binary image uploaded via API; no direct GCS access from mobile |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-03-01 | Tampering | PUT /chef/menu/items/:id price | accept | API creates ApprovalRequest for price changes — admin review required. Client-side banner informs user, server enforces the gate. |
| T-03-03-02 | Information Disclosure | Earnings screen shows bank details | mitigate | Display only masked bank account (last 4 digits) — full details not shown. API should return masked values — if not, truncate on client: `'****' + account.slice(-4)` |
| T-03-03-03 | Injection | Review reply free text | mitigate | Zod min-length validation (10 chars) on client; server stores and returns raw text — rendering uses React Native Text (no HTML injection risk in native) |
| T-03-03-04 | Tampering | Photo upload (profile, kitchen, menu) | mitigate | All uploads use authenticated Bearer token endpoints; no unsigned upload URLs exposed to client |
</threat_model>
