# Quick Task 260617-mp6 — lucide-react-native type errors (vendor)

## Root cause (not what the handoff assumed)
The ~48 vendor `tsc` errors were NOT "lucide type drift from SDK-56." They came
from `apps/mobile-vendor/types/expo-modules.d.ts` — a file of hand-written
`declare module` **stubs** for packages "not yet in node_modules" (per its own
header). SDK 56 installed the real packages, but the ambient stub for
`lucide-react-native` declared only 16 icons and **shadowed** the real package
types. Every icon outside that list → `TS2305 no exported member`.

Verified: `import * as L from 'lucide-react-native'` resolved to a namespace of
exactly those 16 stub icons; `traceResolution` confirmed the real
`dist/lucide-react-native.d.ts` (which DOES export all icons) was being
overridden by the ambient declaration.

All 5 stubbed packages are now installed with real types:
lucide-react-native 0.475.0, expo-image-picker 56, expo-document-picker 56,
expo-haptics 56, @gorhom/bottom-sheet 5.2.14. The whole stub file is obsolete.

## Task 1 — delete the obsolete stub file
- **files:** `apps/mobile-vendor/types/expo-modules.d.ts` (delete)
- **action:** `git rm` the file. Types-only (no runtime impact); real packages
  provide both runtime and complete types.
- **verify:** vendor `tsc --noEmit`: 48 → 5 errors; the 43 lucide TS2305 gone;
  removing the 4 other stubs surfaced ZERO new errors (app usage matches real types).
- **done:** no lucide errors; only pre-existing unrelated errors remain.

## Out of scope (documented)
The 5 remaining errors are pre-existing `expo-router` TS2345 on `router.push(...)`
(e.g. `/support/new`), caused by a **stale `.expo/types/router.d.ts`** (it
predates those route files). That file is auto-generated by expo-router at
build/prebuild — must not be hand-edited; it regenerates on the next native
build. Confirmed present on `main` independent of this change. Separate follow-up.

Customer/delivery apps have no such stub (0 lucide errors) — vendor-only fix.
