# Summary — 260617-mp6

## What
Deleted `apps/mobile-vendor/types/expo-modules.d.ts` (−139 lines, types-only).

## Root cause (corrected the handoff's assumption)
The ~48 errors were not "lucide type drift from SDK 56." The vendor app carried a
file of hand-written `declare module` **stubs** for packages "not yet in
node_modules." SDK 56 installed the real packages, but the ambient stub for
`lucide-react-native` declared only **16 icons** and shadowed the real package's
types — so every icon outside that list raised `TS2305 no exported member`.

Diagnosis trail:
- `import * as L from 'lucide-react-native'` → namespace had exactly the 16 stub
  icons (Camera, Settings, LayoutDashboard, …).
- `tsc --traceResolution` → the real `dist/lucide-react-native.d.ts` (exports all
  icons) was resolved but overridden by the project's ambient `declare module`.
- All 5 stubbed packages are now installed with real types
  (lucide 0.475.0, expo-image-picker 56, expo-document-picker 56, expo-haptics 56,
  @gorhom/bottom-sheet 5.2.14) → the entire stub file is obsolete.

## Verification
- Vendor `tsc --noEmit`: **48 → 5** errors. All 43 lucide `TS2305` gone.
- Removing the other 4 stubs surfaced **zero** new errors (app usage matches the
  real types).
- Customer/delivery apps: 0 lucide errors (no such stub) — vendor-only fix.
- Types-only change; no runtime impact (real packages provide runtime).

## Remaining 5 errors — pre-existing, out of scope
`expo-router` `TS2345` on `router.push(...)` (e.g. `/support/new`,
`/support/${id}`). Cause: stale auto-generated `.expo/types/router.d.ts` that
predates those route files (verified it lacks `/support/new`). That file is
generated by expo-router at build/prebuild and must not be hand-edited — it
regenerates on the next native build. Confirmed present on `main` independent of
this change. Separate follow-up if a green `tsc` is wanted pre-build.

## Notes
- Mobile-only TS change — no service redeploy.
