## Findings to surface (not fixed here — test-only PR)
1. **`already_deleted` branch is unreachable.** `DeleteMyAccount` looks up the
   user with default-scoped `database.DB.First`, which filters soft-deleted rows,
   so a retried delete returns 404 instead of the intended idempotent 200. Test
   documents the current 404 behaviour; recommend `.Unscoped()` on the lookup.
2. **`ChefProfile` has no `gorm.DeletedAt`** → the chef profile is HARD-deleted
   (only `user` is soft-deleted). Behaviour is intentional (kitchen vanishes
   immediately) but worth noting against the "retention" framing.
3. Minor: the handler's `log.Printf` prints the deleted user's email to stdout
   (server-side log PII). Audit row itself is PII-safe. Flag only.
