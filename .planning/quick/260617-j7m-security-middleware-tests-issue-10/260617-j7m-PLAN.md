<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mobile client → API | Untrusted version headers gate (426); spoofable but only self-limits the client |
| internet → API (unauth) | Per-IP rate limit guards credential stuffing / enumeration |
| payment provider → API webhook | Untrusted POST body; authenticity established ONLY by HMAC signature |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-j7m-01 | Spoofing | Razorpay/Stripe webhook | mitigate | Tests pin that a forged/absent signature is rejected and only a correct HMAC (constant-time compare) is accepted — regression guard on VerifyWebhookSignature/VerifyStripeWebhookSignature |
| T-j7m-02 | Tampering (replay) | Stripe webhook | mitigate | Test asserts a stale (now-1h) timestamp is rejected, pinning the 5-min replay tolerance |
| T-j7m-03 | DoS | RateLimit / RateLimitRedis | mitigate | Tests pin 429-after-budget; fail-open behavior is intentional (availability > burst risk) and asserted so it can't silently flip to fail-closed |
| T-j7m-04 | Tampering | Idempotency replay | mitigate | Test asserts the handler executes exactly once on a repeated key, preventing duplicate order/payment side effects |
| T-j7m-05 | Spoofing | VersionCheck headers | accept | Version headers are client-controlled; the wall is a UX nudge, not a security control — no server secret depends on it |
</threat_model>
