# Home Chef — Design Context

This file is the single source of truth for design decisions across all seven Home Chef frontends (3 mobile + 4 web). It is loaded by `/audit`, `/polish`, `/normalize`, and every other Impeccable design skill. Keep it short, opinionated, and current.

## Design Context

### Users

Home Chef serves three distinct audiences, each on their own app, all sharing the same brand:

- **Customers** (`apps/web`, `apps/mobile-customer`) — Hungry people browsing chefs and ordering food on a phone. Decision-fatigued, often in distracted contexts (commute, sofa, hunger). They want appetite, trust, and speed-to-checkout. The interface must make food look great and the order feel safe.
- **Home chefs / vendors** (`apps/vendor-portal`, `apps/mobile-vendor`) — Independent home cooks running a small kitchen-as-a-business from their phone or laptop. Hands often messy, time-pressured between orders. They want efficiency, clear order queues, and pride in their listing. The interface must read fast in a kitchen and never get in the way.
- **Drivers** (`apps/delivery-portal`, `apps/mobile-delivery`) — Delivery riders moving through traffic, glancing at the screen between stops. They want one thing on screen at a time: where to go next. The interface must be glanceable, high-contrast, and never quiz the driver.
- **Admins** (`apps/admin-portal`) — Internal ops monitoring the platform. Dense data, multi-window, desktop-first. They want clarity and signal density, not delight.

**Job to be done:** Connect a hungry person with a home cook nearby and a driver to bring it. The platform succeeds when all three roles trust each other through the app.

### Brand Personality

**Three words: Confident · Appetizing · Quietly modern.**

- **Confident**, not loud — the platform is established enough to not need urgency tricks, countdown timers, or red badges everywhere.
- **Appetizing**, not artisanal — food photography and generous space do the work; we don't lean on terracotta-and-cream "homemade" kitsch.
- **Quietly modern**, not trendy — restrained type, consistent rhythm, one accent. The kind of interface that ages well, not the one that screams "designed in 2025."

Emotional goals: hunger, trust, calm. We are explicitly NOT chasing FOMO, gamified streaks, or aggressive consumer-grade hype.

### Aesthetic Direction — Refined consumer (Stripe Atlas / Cash App / Apple Food)

Bright, clean palette. Photo-forward but composed. Geometric typography. Subtle micro-interactions. The visual reference is closer to Cash App's restraint, Stripe's confidence, or Apple Food's editorial weight — not to Swiggy/Zomato's saturated chaos or to DoorDash's red-tape urgency.

**Palette: Paper · Ink · Persimmon (light-first, dark supported).** (Pivoted from herb-green to persimmon-orange 2026-05-25. Mobile Tailwind token name kept as `herb` to avoid a 49-file rename, but renders persimmon orange. Rename is tracked as TODO.)

- **Paper** — `oklch(0.985 0.003 80)` — slightly warm off-white. Page background. Never pure `#fff`.
- **Ink** — `oklch(0.18 0.01 80)` — near-black with the faintest warm tint. Primary text and primary CTA. Never pure `#000`.
- **Persimmon** — `#C2410C` (≈ `oklch(0.56 0.18 38)`) — deep warm editorial orange. **The single brand accent.** Used for primary actions, focus, success, and selected states. Reads "appetizing, confident, autumnal" — distinct from the red-saturated delivery-app category. Hover: `#9A3412`. Tint (backgrounds): `#FFEDD5`.
- **Bone** — `oklch(0.97 0.005 80)` — surface elevated above paper (cards, popovers, sheets). Subtle, not white.
- **Mist** — `oklch(0.92 0.005 80)` — hairline borders, dividers, muted backgrounds.
- **Functional-only** — Destructive `#B22B0E` (deep red, never confused with persimmon brand orange), `Amber` (`oklch(0.74 0.13 75)`) for warning. These never decorate — only signal.

**Dark mode mirrors this with paper→`oklch(0.16 0.005 80)`, ink→`oklch(0.96 0.003 80)`, persimmon shifts to `#E26542` for AA contrast on dark surfaces.**

**Anti-references — explicitly avoid:**

- The previous terracotta-cream-amber artisanal aesthetic (legacy, migrate as you touch it)
- Red-saturated food-delivery look (Swiggy / Zomato / DoorDash)
- AI slop palette (cyan-on-dark, purple-to-blue gradients, neon accents)
- 2018 indie ecommerce (Playfair Display + cursive accent + sage)
- Web3 / glassmorphism / glowing-edge maximalism
- Generic SaaS dashboard (Linear/Stripe-dashboard-clone)
- Hero metrics + gradient text + same-size card grids

### Typography

Two geometric sans-serifs — **no serifs, no cursive accent fonts.**

- **Display: Geist Sans** (Vercel, free, variable) — headlines, marketing moments, large numerals, brand. Tightly tracked. Used 600/700 weight only.
- **Body / UI: Inter Variable** — body, labels, forms, navigation. Used 400 (body), 500 (emphasis), 600 (headings).
- **Numerals:** Tabular figures (`font-variant-numeric: tabular-nums`) for any price, order ID, ETA, or stat — never proportional figures for numbers that align in columns.
- **Scale:** Fluid `clamp()` with a 1.2 ratio for mobile / 1.25 for desktop. No "display-2xl" 4.5rem chest-thumpers — biggest type is ~2.5rem on mobile, ~3rem on desktop.
- **Avoid:** Playfair Display, Caveat, Open Sans, Roboto, Arial, system-ui-only stacks, ALL CAPS for body, gradient text on headings.

### Spatial System

- **Radius:** 8px default (`--radius: 0.5rem`). Sheets / modals 16px. **No 24-32px super-rounded "friendly" pills.**
- **Shadow scale:** Three steps only (`--shadow-1` hairline, `--shadow-2` card lift, `--shadow-3` modal). Never decorative; only conveys elevation.
- **Hairlines, not bordered cards.** Sections separated by `1px solid var(--mist)`, not by stacking card-in-card.
- **Spacing rhythm:** 4 / 8 / 12 / 16 / 24 / 32 / 48 / 64 / 96 px. Use fluid `clamp()` between viewports.
- **Touch targets:** 44×44 minimum on mobile (driver app: 48×48, gloves and motion).

### Motion

- **Easing:** `cubic-bezier(0.22, 1, 0.36, 1)` (ease-out-quart) for entrances; `cubic-bezier(0.4, 0, 0.2, 1)` for state changes. **No bounce, no elastic, no overshoot.** The existing `bounce-in` keyframe must go.
- **Duration:** 150ms (micro), 250ms (default), 400ms (page transition). Driver app halves all durations for glanceability.
- **What animates:** `opacity` and `transform` only. Layout properties (`width`, `height`, `top`, `left`, `padding`) never animate.
- **Reduced motion:** `prefers-reduced-motion: reduce` removes all transitions, replaces with instant state swaps. Mandatory.

### Component Strategy

- **Form primitives** — already on `@tesserix/web` (Radix + shadcn). Don't reinvent.
- **Brand-level reusable components** (OrderCard, ChefCard, DriverHandoff, MoneyPill, OrderStatusStepper) should live in `packages/mobile-shared/` for native and a future shared web package. Currently many are duplicated across the four web portals — extracting them is a `/extract` task, not for this audit.
- **App-specific compositions** stay in `apps/{app}/src/components/`. Promote when a second app needs them.

### Accessibility

- **WCAG 2.1 AA baseline on every surface.** AAA where practical for body text.
- Skip link is the first focusable element on every web page.
- Semantic landmarks (`<main>`, `<nav>`, `<header>`) on every screen, semantic HTML over `div` always.
- Visible **persimmon** focus ring (`2px solid var(--herb)` — token name retained, value is now persimmon orange — with 2px offset) — never `outline: none`.
- `prefers-reduced-motion` honored everywhere.
- Driver app has its own contrast floor (7:1 minimum, all type ≥16px) — driver context is safety-critical.
- Screen-reader-friendly form errors (`aria-describedby`, `aria-invalid`).

### Design Principles (the 5 that override everything)

1. **Photo-forward, chrome-light.** Food and faces carry the brand. UI chrome shrinks to make room — hairlines beat borders, sans beats serif, paper beats card-soup.
2. **One accent, used sparingly.** Persimmon orange is for primary action, focus, and success — nothing decorative. If a screen has more than one persimmon element competing for the eye, one of them is wrong.
3. **Confidence through restraint.** No bounce easing, no glassmorphism, no gradient text, no decorative shadows, no hero metric layouts. If it feels loud, it's wrong.
4. **Per-role density, shared language.** Customer apps breathe, vendor apps pack more information, driver apps strip down to one glanceable task. The *tokens* are shared; the *layouts* adapt.
5. **Trust the system.** Use the tokens, use the `@tesserix/web` primitives, use Geist + Inter, use the easing curve. Don't invent per-feature colors, radii, or motion.

## Vendor v2 surface model (owner pivot, 2026-06-10)

Owner feedback: the flat hairline-minimal execution reads "basic and outdated." For the
**vendor app** (and as the direction for the other mobile apps as they're touched), the
"hairlines, not bordered cards" rule is replaced by a **canvas + cards** model:

- Page canvas: `bone` (#F5F5F4). Content in white `paper` cards, radius 16, shadow tokens 1/2.
- Status communicated by tint-bg chips, not bare dots. Grouped lists are iOS-style white
  rounded card groups with inset hairlines, not edge-to-edge page hairlines.
- Buttons: ink-filled primary + bordered ghost secondary (radius 12, ≥48pt). No naked
  underlined text actions. Links are persimmon, no underline.
- Segmented controls: mist track + white active segment with shadow 1 (no bare-text underline tabs).
- Tab bar: white, top-elevated, active item persimmon (selected-state accent is sanctioned).

Everything else holds: palette, Geist/Inter, single-accent discipline, motion rules, a11y.
Canonical spec: `.planning/quick/260610-ica-vendor-app-v2-visual-redesign-canvas-car/260610-ica-UI-V2-SPEC.md`.

## What this replaces

The existing `apps/web/src/styles/globals.css` defines a terracotta-cream-amber Playfair-Caveat artisanal system with `bounce-in`, `glass`, `card-hover-lift`, and decorative gradients. **All of that is legacy.** The `/audit` will flag specific instances; `/normalize` will migrate them in waves. Until migration is complete, treat new screens as the source of truth and old screens as work-in-progress.

## How to use this file

- Every design skill (`/audit`, `/polish`, `/normalize`, `/animate`, `/quieter`, `/distill`, etc.) reads this file as context.
- When the user pushes back on a design choice, update this file first, then re-run the skill.
- Never embed brand colors or fonts directly in code — they must live in CSS variables / Tailwind config and trace back to this file.
